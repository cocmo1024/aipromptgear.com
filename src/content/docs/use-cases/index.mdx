# Use Cases

Start here when the real question is not "which model should we use?" but "what kind of work are we trying to improve?" This section keeps coverage anchored to teams, outcomes, and operating constraints before technology choices enter the picture.

<div class="signal-strip">
	<div>
		<strong>Best for</strong>
		<p>Support, research, operations, knowledge work, and other repeatable processes with clear review boundaries.</p>
	</div>
	<div>
		<strong>Avoid this mistake</strong>
		<p>Do not jump straight into prompts or models until the workflow owner, inputs, outputs, and review point are clear.</p>
	</div>
	<div>
		<strong>What comes next</strong>
		<p>After the use case is framed, move into workflows, then models, then tooling and evaluation.</p>
	</div>
</div>

## Decision flow

<Steps>
	1. Define the job to be done, the team owner, and the output that must improve.
	2. Identify the sources of truth, required guardrails, and whether a human reviewer must approve output.
	3. Decide where the prompt system should assist, draft, route, or decide.
	4. Move into workflow and model design only after those boundaries are written down.
</Steps>
