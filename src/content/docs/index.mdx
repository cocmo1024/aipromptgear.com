## What this site optimizes

<div class="signal-strip">
	<div>
		<strong>Research model</strong>
		<p>Application first, workflow second, model and tooling last. This keeps coverage grounded in real operating work.</p>
	</div>
	<div>
		<strong>Decision value</strong>
		<p>The strongest pages help readers decide how a workflow, model route, tool boundary, or software purchase should actually work.</p>
	</div>
	<div>
		<strong>Long-term edge</strong>
		<p>Reference pages can be reviewed and updated without being rewritten from scratch whenever models, tools, or costs shift.</p>
	</div>

## How to use this reference

<Steps>
	1. Start with the team or workflow category that matches the operating problem.
	2. Move into workflow design to shape prompts, steps, escalation rules, and handoffs.
	3. Use models and APIs to pressure-test fit, latency, cost, and deployment boundaries.
	4. Use tooling and evaluation to turn a promising prompt flow into a maintainable operating system.
</Steps>
